# Firewall Ticket Investigation SOP  
## Step-by-Step Operational Workflow (SOC Oriented)

---

## 🔹 Purpose

This document outlines the structured approach I follow while handling firewall-related tickets in a SOC environment.  

The objective is to ensure consistent investigation, proper threat validation, accurate escalation, and disciplined case closure.

This workflow helps reduce false positives and ensures no malicious activity is overlooked.

---

# Step 1 – Alert Identification & Initial Review

When a firewall alert ticket is assigned:

1. Confirm the alert is generated by the firewall device.
2. Review ticket subject and description carefully.
3. Identify:
   - Source IP
   - Destination IP
   - Destination Port
   - Protocol (TCP/UDP)
   - Timestamp
   - Firewall Action (Allowed / Denied / Dropped)
4. Check severity level assigned by the system.

At this stage, I try to understand:

- Is this inbound or outbound traffic?
- Is this a single event or repeated activity?
- Does this look like normal traffic?

---

# Step 2 – Source IP Validation

## 2.1 Internal Asset Verification

- Check asset inventory to confirm whether source IP belongs to internal network.
- If internal:
  - Identify asset owner
  - Check asset type (Server / Workstation / Application Node)

## 2.2 External Source Analysis

If source is external:

- Perform reputation lookup using threat intelligence platforms.
- Check:
  - Geolocation
  - ISP / ASN
  - Previous abuse reports
  - Blacklist status
  - TOR or proxy usage

Important Note:
IP reputation alone does not confirm malicious intent. It supports investigation.

---

# Step 3 – Destination & Port Analysis

1. Identify which system is being targeted.
2. Confirm if destination system is:
   - Public-facing
   - Internal server
   - Critical production asset

3. Validate destination port and associated service.

Examples:
- 53 → DNS
- 445 → SMB
- 3389 → RDP
- 80/443 → Web services

Ask:
- Is access to this port expected from this source?
- Does this align with normal business operations?

---

# Step 4 – Firewall Action & Rule Verification

1. Check whether traffic was allowed or blocked.
2. Identify which firewall rule triggered.
3. Review session logs:
   - Number of connection attempts
   - Session duration
   - Packet volume
   - Repeated pattern frequency

If blocked:
- Validate whether rule functioned correctly.

If allowed:
- Proceed to deeper correlation.

---

# Step 5 – Traffic Pattern & Behavior Analysis

At this stage, I analyze behavioral indicators.

Look for:

- Multiple rapid connection attempts → Possible brute force
- High volume UDP traffic → Possible DDoS
- Sequential port access → Port scanning
- Repeated SMB attempts → Lateral movement attempt
- Large outbound traffic → Data exfiltration

Compare with historical baseline if available.

---

# Step 6 – Cross-Log Correlation

If traffic was allowed or suspicious:

1. Check SIEM for related alerts.
2. Review IPS logs for exploit detection.
3. Check endpoint logs for:
   - Failed logins
   - Privilege escalation
   - Suspicious process execution
4. Validate authentication logs.
5. Check outbound communication attempts to suspicious IP.

Goal:
Determine whether activity is isolated or part of multi-stage attack.

---

# Step 7 – Threat Classification

Based on full analysis, classify ticket as:

- False Positive
- Expected Business Activity
- Suspicious but Unconfirmed
- Confirmed Malicious Activity

Assign Risk Level:
Low / Medium / High / Critical

Escalate to L2 / IR team if required.

---

# Step 8 – Remediation & Containment

If malicious activity confirmed:

- Coordinate with Network Team to block IP (inbound & outbound).
- Confirm rule enforcement.
- Monitor for recurrence.
- Recommend firewall rule tuning if required.

For DoS/DDoS Scenarios:

1. Confirm if attack is ongoing.
2. Inform Network Team immediately.
3. If required, escalate to ISP for mitigation support.
4. Continue monitoring after mitigation.
5. Ensure traffic stability before closing ticket.

---

# Step 9 – Documentation & Closure

Before closing ticket:

- Document investigation steps clearly.
- Mention intelligence findings.
- Record firewall action taken.
- Note business impact.
- Provide preventive recommendation.
- Confirm no further suspicious activity observed.

Case is closed only after validation is complete.

---

# Investigation Principles I Follow

- Never rely on reputation alone.
- Always validate context and baseline.
- Correlation increases investigation confidence.
- Documentation must be clear for audit trail.
- Every alert deserves structured validation.

---

# Objective of This Workflow

To maintain consistent, professional, and risk-based handling of firewall alerts aligned with SOC best practices and company operational standards.
